# Git Commit Convention

## Purpose

Commit messages are a permanent engineering record.

They should communicate:

* Why a change exists
* What problem is being solved
* The architectural or business intent behind the change

Commit messages are not a log of editor activity or implementation details.

This repository follows the Conventional Commits specification with additional engineering rules.

---

# Commit Message Format

```text
<type>(<scope>): <subject>

[optional body]

[optional footer]
```

Example:

```text
feat(rag): add hybrid retrieval support

Introduce keyword + vector search to improve
retrieval quality for low-context queries.

Impact:
- requires reindexing existing documents
```

---

# Commit Types

## feat

New functionality.

```text
feat(chat): support streaming responses
```

---

## fix

Bug fix.

```text
fix(auth): prevent refresh token reuse
```

---

## refactor

Internal code improvement without behavior changes.

```text
refactor(api): extract tenant resolver
```

---

## perf

Performance improvement.

```text
perf(qdrant): reduce query latency
```

---

## docs

Documentation only.

```text
docs(architecture): add retrieval workflow
```

---

## test

Tests only.

```text
test(chat): add websocket integration tests
```

---

## build

Build system or dependency packaging changes.

```text
build(docker): upgrade node runtime
```

---

## ci

CI/CD changes.

```text
ci(gitlab): add deployment pipeline
```

---

## chore

Maintenance work.

```text
chore(deps): update dependencies
```

---

## style

Formatting only.

```text
style(api): apply prettier formatting
```

---

## revert

Revert a previous commit.

```text
revert(auth): remove token rotation
```

---

Examples:


```text

```

```text

```

# Subject Rules

The subject line must:

* use imperative mood
* be lowercase
* contain no trailing period
* be concise
* describe intent

Good:

```text
feat(rag): add hybrid retrieval strategy
```

Bad:

```text
feat(rag): Added hybrid retrieval strategy.
```

Bad:

```text
feat(rag): changes
```

Bad:

```text
update stuff
```

---

# Commit Body Guidelines

Add a body when the reason for the change is not obvious.

Recommended structure:

```text
Why:
- explain the problem

Changes:
- summarize important modifications

Impact:
- deployment effects
- migrations
- compatibility concerns
```

Example:

```text
feat(tenant): implement tenant-aware retrieval

Why:
- prevent document leakage between tenants

Changes:
- add tenant filter abstraction
- update qdrant queries
- add integration tests

Impact:
- requires collection migration
```

---

# Breaking Changes

Breaking changes must be explicitly marked.

Example:

```text
feat(api)!: redesign authentication endpoints
```

or

```text
feat(api): redesign authentication endpoints

BREAKING CHANGE:
clients must migrate to v2 endpoints
```

---

# Commit Granularity

A commit should represent one logical change.

Preferred:

```text
feat(auth): add refresh token rotation
```

```text
docs(auth): document refresh token flow
```

Avoid:

```text
feat(auth): add refresh token rotation

also:
- update docker
- fix nginx
- rename variables
- update README
```

If a change can be reverted independently, it should usually be a separate commit.

---

# AI-Assisted Development

AI tools may generate code, but commit messages must describe the engineering intent.

Do not mention tools in commit messages.

Avoid:

```text
feat(api): generated by cursor
```

```text
fix(chat): chatgpt suggestion
```

```text
refactor(rag): ai improvements
```

Instead describe the actual change:

```text
refactor(rag): isolate retrieval pipeline
```

# Pull Request Rule

A pull request should answer:

1. Why is this change needed?
2. What behavior changed?
3. What risks were introduced?
4. How can it be validated?

Commit messages should provide enough context that future engineers can understand repository history without reading every pull request.

---

# Golden Rule

Every commit should make sense when read six months later by someone unfamiliar with the implementation details.

Optimize commit messages for future understanding, not for the moment of writing.
