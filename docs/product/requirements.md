# Cogence Requirements

## Overview

This document defines the functional and non-functional requirements for Cogence MVP v1.

**Primary Goal:** Generate daily business-readable reports from engineering activity.

**Success Metric:** A non-technical manager can understand what engineering accomplished in less than 60 seconds.

---

## Functional Requirements

### FR-1: Data Collection

#### FR-1.1: Gitea Integration
- **Must** connect to Gitea instance using API token
- **Must** discover all accessible repositories
- **Must** fetch commit metadata from repositories
- **Must** handle API rate limiting gracefully
- **Must** retry failed API requests with exponential backoff

#### FR-1.2: Commit Data Collection
- **Must** collect the following for each commit:
  - Repository name
  - Commit SHA
  - Author name
  - Author email
  - Timestamp
  - Commit message title
  - Commit message body (description)
  - Files changed count
  - Lines added (insertions)
  - Lines deleted (deletions)

#### FR-1.3: Scheduled Collection
- **Must** run data collection on a schedule (per ADR-011)
- **Must** collect commits from the last 24 hours
- **Must** avoid duplicate commit collection
- **Must** log collection status and errors

### FR-2: Data Storage

#### FR-2.1: Database Schema
- **Must** store commit metadata in PostgreSQL (per ADR-010)
- **Must** maintain referential integrity between repositories and commits
- **Must** index frequently queried fields (timestamp, repository, author)
- **Must** support efficient date-range queries

#### FR-2.2: Data Retention
- **Must** retain commit data for at least 90 days
- **Should** implement data archival strategy for older data
- **Must** never delete data without explicit user action

### FR-3: Report Generation

#### FR-3.1: Daily Report
- **Must** generate a daily report for the last 24 hours
- **Must** include the following sections:
  - Executive Summary
  - Projects Worked On
  - Contributor Summary
  - Management Notes
- **Must** use business language (per ADR-002)
- **Must** be readable in under 60 seconds (per ADR-007)

#### FR-3.2: Executive Summary
- **Must** provide 2-4 sentence overview of engineering activity
- **Must** highlight key accomplishments
- **Must** use non-technical language
- **Must** be generated by AI (per ADR-005)

#### FR-3.3: Projects Worked On
- **Must** list all repositories with activity
- **Must** provide brief description of work per repository
- **Must** group related commits by repository

#### FR-3.4: Contributor Summary
- **Must** list all contributors with activity
- **Must** provide brief summary of each contributor's work
- **Must NOT** rank or score contributors (per ADR-004)
- **Must NOT** include productivity metrics

#### FR-3.5: Management Notes
- **Must** highlight important observations
- **Must** identify potential risks or concerns
- **Should** suggest areas requiring leadership attention
- **Must** be actionable (per principle #8)

### FR-4: Report Delivery

#### FR-4.1: Report Access
- **Must** provide API endpoint to retrieve daily reports
- **Must** support querying reports by date
- **Must** return reports in JSON format
- **Should** support HTML format for email delivery

#### FR-4.2: Report Storage
- **Must** store generated reports in database
- **Must** cache reports to avoid regeneration
- **Must** support report regeneration if needed

### FR-5: Configuration

#### FR-5.1: System Configuration
- **Must** support configuration via environment variables
- **Must** require Gitea URL and access token
- **Should** support optional company name
- **Must** validate configuration on startup

#### FR-5.2: Repository Configuration
- **Should** support repository inclusion/exclusion filters
- **Should** support repository grouping (e.g., by team)
- **Must** default to including all accessible repositories

## Non-Functional Requirements

### NFR-1: Performance

#### NFR-1.1: Data Collection
- **Must** complete data collection within 5 minutes for up to 100 repositories
- **Must** handle up to 10,000 commits per day
- **Should** process commits in parallel where possible

#### NFR-1.2: Report Generation
- **Must** generate daily report within 30 seconds
- **Must** support concurrent report requests
- **Should** cache frequently accessed reports

#### NFR-1.3: API Response Time
- **Must** respond to API requests within 2 seconds (95th percentile)
- **Must** handle at least 10 concurrent requests

### NFR-2: Reliability

#### NFR-2.1: Availability
- **Must** achieve 99% uptime during business hours
- **Must** gracefully handle external service failures (Gitea, LLM)
- **Must** implement health check endpoints

#### NFR-2.2: Data Integrity
- **Must** ensure commit data accuracy
- **Must** validate all data before storage
- **Must** implement database transactions for related operations
- **Must** log all data collection and processing errors

#### NFR-2.3: Error Handling
- **Must** provide meaningful error messages
- **Must** log errors with sufficient context for debugging
- **Must** implement retry logic for transient failures
- **Should** alert on critical failures

### NFR-3: Security

#### NFR-3.1: Authentication
- **Must** secure API endpoints with authentication
- **Must** store Gitea tokens securely (encrypted at rest)
- **Must** support token rotation
- **Must** implement rate limiting on API endpoints

#### NFR-3.2: Data Privacy
- **Must** respect user privacy (per ADR-004)
- **Must NOT** expose individual developer metrics publicly
- **Must** implement access controls for sensitive data
- **Must** comply with data protection regulations

#### NFR-3.3: Audit Logging
- **Must** log all data access and modifications
- **Must** log authentication attempts
- **Should** implement audit trail for report generation

### NFR-4: Maintainability

#### NFR-4.1: Code Quality
- **Must** follow PEP 8 style guidelines
- **Must** include type hints for all functions
- **Must** achieve 80%+ test coverage
- **Must** document all public APIs

#### NFR-4.2: Monitoring
- **Must** implement application logging
- **Must** expose metrics for monitoring (response times, error rates)
- **Should** integrate with monitoring tools (Prometheus, Grafana)

#### NFR-4.3: Database Migrations
- **Must** use Alembic for schema migrations
- **Must** support forward and backward migrations
- **Must** test migrations before deployment

### NFR-5: Scalability

#### NFR-5.1: Data Volume
- **Must** support up to 100 repositories
- **Must** handle up to 10,000 commits per day
- **Should** design for future growth to 1,000 repositories

#### NFR-5.2: User Load
- **Must** support up to 50 concurrent users
- **Should** design for horizontal scaling

### NFR-6: Usability

#### NFR-6.1: Report Readability
- **Must** use clear, concise language
- **Must** avoid technical jargon
- **Must** structure reports for quick scanning
- **Must** highlight key information

#### NFR-6.2: API Usability
- **Must** provide clear API documentation
- **Must** use RESTful conventions
- **Must** return structured error responses
- **Should** provide API examples

## Constraints

### Technical Constraints
- **Must** use Python 3.11+
- **Must** use FastAPI framework (per ADR-009)
- **Must** use PostgreSQL database (per ADR-010)
- **Must** integrate with Gitea API only (MVP scope)
- **Must** use LLM for summary generation (per ADR-005)

### Business Constraints
- **Must** be single-tenant (per ADR-008)
- **Must** be deployed internally first
- **Must** focus on daily reports only (per ADR-006)
- **Must NOT** implement code analysis (per ADR-003)

### Time Constraints
- **Must** deliver MVP v1 within defined timeline
- **Must** prioritize core functionality over advanced features

---

## Assumptions

1. Gitea instance is accessible and stable
2. LLM API is available and reliable
3. Users have basic understanding of Git concepts
4. Reports will be consumed primarily by managers, not engineers
5. Single organization deployment (no multi-tenancy needed)
6. English language support only in MVP

---

## Dependencies

### External Services
- Gitea API (required)
- LLM API (required for summaries)
- SMTP server (optional, for email delivery)

### Infrastructure
- PostgreSQL database server
- Application server (for FastAPI)
- Scheduled task runner (for data collection)

---

## Out of Scope (MVP v1)

The following are explicitly excluded from MVP v1:

- Pull request analysis
- Issue/ticket integration (Jira, GitHub Issues)
- Deployment tracking
- CI/CD integration
- Code quality analysis
- Static code analysis
- Source code embeddings
- Vector databases
- Real-time data collection
- Multi-repository code search
- Developer productivity scoring
- Team performance analytics
- Weekly/monthly reports
- Custom report templates
- Multi-tenancy
- SaaS deployment
- Mobile applications
- Slack/Teams integration

These may be considered for future versions based on MVP validation.

---

## Success Criteria

MVP v1 is successful if:

1. ✅ A manager can read and understand a daily report in under 60 seconds
2. ✅ Reports accurately reflect engineering activity from commits
3. ✅ Reports use business language, not technical jargon
4. ✅ System reliably collects commit data daily
5. ✅ Reports are generated automatically without manual intervention
6. ✅ Users trust the information presented (per principle #7)
7. ✅ System handles errors gracefully without data loss
8. ✅ Reports provide actionable insights for leadership
