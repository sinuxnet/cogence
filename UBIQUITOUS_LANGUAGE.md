# Ubiquitous Language

## Platform

| Term | Definition | Aliases to avoid |
| ---- | ---------- | ---------------- |
| **Engineering Intelligence** | The transformation of engineering signals into business-readable insight | Analytics, metrics, monitoring |

## Core entities

| Term | Definition | Aliases to avoid |
| ---- | ---------- | ---------------- |
| **Organization** | The company using a single Cogence deployment | Tenant, customer, account |
| **Repository** | A Git repository tracked as a source of commit signals | Project, codebase, repo |
| **Commit** | A Git commit with metadata (SHA, author, timestamp, message) | Change, update, modification |
| **Contributor** | A person who authors commits, identified by email | Developer, engineer, programmer |
| **Daily Report** | A generated summary of engineering activity for one calendar day | Dashboard, analytics, metrics |
| **Summary** | AI-generated business-readable text derived from commit data | Description, translation, explanation |
| **Signal** | A data point from engineering activity used for insight | Metric, measurement, data point |
| **Activity** | Engineering work represented by commits within a period | Productivity, output, throughput |

## Report composition

| Term | Definition | Aliases to avoid |
| ---- | ---------- | ---------------- |
| **Executive Summary** | A 2–4 sentence overview of the day's engineering activity | TL;DR, headline, abstract |
| **Repository Summary** | A brief business-language description of work in one repository | Project summary, repo blurb |
| **Contributor Summary** | A neutral description of one contributor's work that day | Developer summary, performance note |
| **Management Notes** | Observations and recommendations for leadership | Alerts, warnings, flags |
| **Calendar Day** | One full day in `Asia/Tehran` (00:00–23:59) that a report covers | Last 24 hours, rolling window |
| **Report Date** | The calendar day a Daily Report summarizes | Timestamp, generation date |
| **Observability Gap** | A neutral note when a commit message lacks enough detail for a precise summary | Bad committer, poor hygiene, non-compliant |
| **Report Locale** | The configured language for generated report text (e.g. `fa`, `en`) | Auto-detected language, mixed locale |
| **Report Depth** | How much source material the LLM reads: `brief`, `standard`, or `deep` | Report size, detail level, verbosity |

## Generation inputs

| Term | Definition | Aliases to avoid |
| ---- | ---------- | ---------------- |
| **Truncated Diff** | A capped unified diff fetched at generation time and discarded afterward | Full diff, patch, code snapshot |
| **Atomic Commit** | A commit small enough in scope to summarize precisely | Good commit, clean commit |
| **Non-atomic Commit** | A commit exceeding the file-change threshold, reducing summary precision | Large commit, messy commit, bad commit |

## Processes

| Term | Definition | Aliases to avoid |
| ---- | ---------- | ---------------- |
| **Collection** | Fetching commits from Gitea and storing them as the system of record | Scraping, pulling, syncing |
| **Generation** | Creating a Daily Report from stored commits using an LLM | Building, compiling, rendering |
| **Aggregation** | Combining commits by repository or contributor for summarization | Grouping, roll-up |
| **Delivery** | Sending a generated Daily Report via API or Rocket.Chat | Push, notification, broadcast |

## People and readers

| Term | Definition | Aliases to avoid |
| ---- | ---------- | ---------------- |
| **Reader** | A non-technical leader who consumes Daily Reports | User, stakeholder, viewer |
| **Author** | The name and email on a commit's Git metadata | Committer (when distinct from author) |

## Quality attributes

| Term | Definition | Aliases to avoid |
| ---- | ---------- | ---------------- |
| **Business-readable** | Understandable by a non-technical manager without explanation | Plain English, simplified |
| **Actionable** | Providing enough context for a leadership decision | Informative, detailed |
| **Privacy-respecting** | Aggregated insight without individual scoring or ranking | Anonymized, sanitized |

## Relationships

- An **Organization** owns one or more **Repositories**
- A **Repository** contains many **Commits**
- A **Commit** is authored by exactly one **Author**, aggregated into one **Contributor**
- A **Daily Report** covers exactly one **Calendar Day** and summarizes many **Commits**
- A **Daily Report** is generated in one **Report Locale** at one **Report Depth**
- A **Daily Report** includes one **Executive Summary**, zero or more **Repository Summaries**, zero or more **Contributor Summaries**, and one **Management Notes** section
- Each **Repository Summary** and **Contributor Summary** is a **Summary** generated by an LLM from **Commit** data and optional **Truncated Diff** text
- An **Observability Gap** may appear inside a **Contributor Summary** when message detail is insufficient
- **Collection** populates **Commits**; **Generation** fetches **Truncated Diffs**, produces a **Daily Report**, and discards diffs; **Delivery** sends it to **Readers**

## Example dialogue

> **Dev:** "A commit message is just `fix`. How does **generation** still produce a **Contributor Summary**?"
>
> **Domain expert:** "For **standard** **Report Depth**, the LLM reads the **truncated diff** at generation time — not stored, just used once. If the diff is thin too, we add an **Observability Gap** note: neutral, factual, not punitive."
>
> **Dev:** "So we flag the engineer?"
>
> **Domain expert:** "No — we flag the **observability gap**, not the person. **Privacy-respecting** means no 'bad committer' language. The note helps **readers** trust the **summary** when confidence is low."
>
> **Dev:** "What about **report locale** — do we detect Persian from commit messages?"
>
> **Domain expert:** "Never. **Report Locale** is explicit config. The demo needs `fa`; English is `en`. Same **commits**, different **business-readable** output."
>
> **Dev:** "And **non-atomic commits** — do they show up in **management notes**?"
>
> **Domain expert:** "Team-level only. When commits exceed the threshold, **management notes** explain that precision may drop and **atomic commits** help. No rankings, no surveillance."

## Flagged ambiguities

- **Project vs Repository** — "Project" appears in business speech ("customer-facing projects") and legacy code (`projectsSummary`), but the canonical entity is **Repository**. Use **Repository Summary** in code and docs; reserve "project" only when quoting a manager's business initiative, not as a stored entity.
- **Author vs Contributor** — **Author** is raw Git metadata on a single **Commit**; **Contributor** is the derived person identity (by email) across commits. Reports mention **Contributors**; ingestion validates **Authors**. Multiple Git identities for one person remain separate **Contributors** in MVP v1.
- **Report vs Daily Report** — In MVP v1, **Report** always means **Daily Report**. Avoid weekly/monthly "reports" until those types exist.
- **Productivity vs Activity** — Never measure people by volume. **Activity** describes organizational work patterns; "productivity," commit counts in narrative, and rankings are forbidden.
- **Metric vs Signal** — Cogence produces **signals** (commits) and **summaries**, not **metrics**. Avoid LOC, commit counts in narrative text, and rankings.
- **Organization vs customer** — The deploying company is an **Organization** (single-tenant). "Customer" refers to end-users of the business being built, not the Cogence tenant.
- **Observability Gap vs quality judgment** — An **Observability Gap** states missing context; it is not a commit-quality score or cultural enforcement mechanism.
- **Active Repositories vs Repository Summary** — MVP docs label the section "Active Repositories" (repos with activity that day); each entry's text is a **Repository Summary**. Same concept, different granularity (section vs sentence).
